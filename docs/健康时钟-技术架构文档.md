# 健康时钟 App 技术架构文档

## 1. 文档目标

本文档用于明确健康时钟 App 的首版技术架构、服务职责划分、模块边界和推荐实现方式，便于后续让 AI 继续生成数据库设计、接口文档和项目初始化代码。

## 2. 最终选型结论

### 客户端

- Flutter
- iOS 首发

### 服务端

- FastAPI 作为业务后端

### 基础设施

- Supabase Auth 负责认证
- Supabase Postgres 负责主数据库
- Cloudflare R2 负责对象存储

### 登录方式

- 手机号验证码登录
- Apple 登录

## 3. 为什么这样选

### 选择 FastAPI 的原因

- 这个项目的核心能力集中在 AI、OCR、文本抽取和结构化处理
- Python 生态更适合快速接入 AI 服务和文档处理链路
- MVP 阶段更需要迭代速度和开发效率
- FastAPI 结构清晰，适合后续继续拆分服务

### 暂不优先 Go 的原因

- 当前不是高并发事务型系统
- 首版重点不是吞吐极限，而是功能闭环和 AI 能力落地
- 在 Supabase 已经承担认证和数据库能力后，Go 的收益没有 FastAPI 明显

## 4. 总体架构

```text
Flutter App
  |-- Supabase Auth：登录、会话、用户身份
  |-- FastAPI：AI 解析、业务接口、文件上传签名、OCR 编排
  |-- Supabase Postgres：业务数据
  |-- Cloudflare R2：图片、PDF、报告文件
```

## 5. 各系统职责划分

## 5.1 Flutter 客户端

- 用户登录
- 成员切换
- 日历和档案展示
- 发起 AI 创建提醒
- 上传文件
- 展示 OCR/AI 解析结果
- 记录健康指标
- 注册本地通知

## 5.2 Supabase Auth

- 手机号验证码登录
- Apple 登录
- 用户会话管理
- 用户身份校验

## 5.3 Supabase Postgres

- 用户表
- 成员档案表
- 健康提醒表
- 文件元数据表
- 健康指标记录表
- AI 解析记录表

## 5.4 FastAPI

- AI 文本提醒解析
- 语音转写结果的结构化解析
- OCR 文本的二次清洗和结构化
- 从报告中提取候选提醒
- 健康文档摘要生成
- 上传前签名与上传后回调处理
- 统一业务鉴权和规则校验

## 5.5 Cloudflare R2

- 原始图片存储
- PDF 存储
- 就医记录附件存储
- 文件访问路径管理

## 6. 推荐数据流

## 6.1 文本创建提醒

1. Flutter 提交自然语言输入
2. FastAPI 调用 AI 解析为结构化事件
3. FastAPI 返回提醒预览
4. 用户确认后写入 Supabase Postgres
5. 客户端注册本地通知

## 6.2 上传文档生成提醒

1. Flutter 请求 FastAPI 获取上传签名
2. Flutter 上传文件到 R2
3. FastAPI 保存文件元数据
4. FastAPI 执行 OCR 和 AI 结构化抽取
5. 返回候选提醒给客户端确认
6. 确认后写入事件表和文档表

## 6.3 记录健康数据

1. Flutter 提交指标数据
2. FastAPI 做规则校验
3. 写入 Supabase Postgres
4. 客户端刷新档案和趋势图

## 12. 风险与控制

### 风险 1：FastAPI 与 Supabase 权限边界不清晰

控制建议：

- 认证交给 Supabase
- 业务规则交给 FastAPI
- 数据权限通过用户 ID 和成员归属双重校验

### 风险 2：R2 文件上传链路复杂

控制建议：

- 统一通过 FastAPI 签名
- 所有文件元数据入库
- 删除文件时同步删除元数据

### 风险 3：AI 输出不稳定

控制建议：

- 统一 JSON schema
- 必须展示确认页
- 保留原始输入和置信度

## 14. MVP 安全架构（先上线可用版）

### 14.1 认证与鉴权

- 认证统一由 Supabase Auth 负责
- FastAPI 只信任已校验的 Supabase JWT
- 每个业务接口必须校验 user_id 与 member_id 归属关系
- 默认拒绝跨成员访问，避免家庭数据串读

### 14.2 传输与存储安全

- 全链路仅允许 HTTPS
- R2 文件访问采用短时签名 URL，不暴露长期公开链接
- 数据库中不保存明文敏感凭证（如第三方 API Key）
- 环境变量通过 .env 管理，不写入客户端代码仓库

### 14.3 MVP 安全基线

- 接口参数做基础校验（长度、类型、必填）
- 上传文件限制类型与大小（图片/PDF，建议 <= 10MB）
- 关键操作写审计日志（登录、上传、删除、成员编辑）

## 15. MVP API 规范（最小可执行）

### 15.1 基础约定

- Base Path：`/api/v1`
- 认证头：`Authorization: Bearer <supabase_access_token>`
- 请求体格式：JSON（文件上传接口除外）
- 时间统一使用 ISO 8601（含时区）

### 15.2 统一响应结构

成功响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

失败响应：

```json
{
  "code": 1001,
  "message": "member not found",
  "data": null
}
```

### 15.3 MVP 错误码建议

- 0：成功
- 1001：资源不存在
- 1002：参数错误
- 1003：未认证或 token 无效
- 1004：无权限访问该成员数据
- 2001：AI 解析失败
- 2002：OCR 处理失败
- 3001：文件上传失败

## 16. 第三方服务选型（MVP）

### 16.1 AI 服务

- 首选：单一模型服务商，先保证稳定性与成本可控
- 能力要求：结构化 JSON 输出、中文时间表达解析、低延迟
- 降级策略：AI 失败时回退到手动创建流程

### 16.2 OCR 服务

- 首选方案：iOS 端原生 OCR + 服务端 AI 二次结构化
- 备选方案：云 OCR（在识别质量不足时切换）
- 原文保留：必须保存 OCR 原文，便于人工核对

### 16.3 短信与推送

- 手机验证码：使用 Supabase Auth 提供的短信链路
- 提醒通知：MVP 仅 iOS 本地通知，不依赖服务端推送

## 17. 环境配置（MVP）

### 17.1 环境划分

- dev：本地开发环境
- staging：联调与预发布验证
- prod：线上环境

### 17.2 关键环境变量

- `SUPABASE_URL`
- `SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`（仅服务端）
- `R2_ACCESS_KEY_ID`
- `R2_SECRET_ACCESS_KEY`
- `R2_BUCKET`
- `AI_API_KEY`

### 17.3 配置原则

- 客户端仅持有 anon key，不持有 service role key
- 所有密钥按环境隔离
- 生产环境密钥定期轮换

## 18. 部署方案（MVP）

### 18.1 FastAPI 部署

- 使用 Docker 部署 FastAPI
- 运行方式：`uvicorn + gunicorn`（多 worker）
- 通过反向代理（Nginx/Caddy）提供 HTTPS 入口

### 18.2 发布流程（最小版）

1. 合并到主分支
2. 自动化执行基础测试
3. 构建镜像并发布到 staging
4. 验证核心流程后发布到 prod

### 18.3 回滚策略

- 保留最近可用镜像版本
- 发布失败时直接回滚到上一个稳定版本
- 数据库变更采用可回滚迁移脚本

## 19. 监控与告警（MVP）

### 19.1 必监控指标

- API 成功率
- API P95 响应时间
- AI/OCR 调用失败率
- 文件上传失败率

### 19.2 日志要求

- 接口请求日志（trace_id、user_id、path、status）
- 错误日志（异常堆栈、上下文参数）
- 审计日志（成员删除、文件删除、登录）

### 19.3 MVP 告警规则

- 5 分钟内接口错误率 > 5% 触发告警
- AI/OCR 连续失败超过阈值触发告警
- 文件上传失败率异常升高触发告警
