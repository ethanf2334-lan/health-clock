# 健康时钟 App 技术架构文档

## 1. 文档目标

本文档用于明确健康时钟 App 的首版技术架构、服务职责划分、模块边界和推荐实现方式，便于后续让 AI 继续生成数据库设计、接口文档和项目初始化代码。

## 2. 最终选型结论

### 客户端

- Flutter
- iOS 首发

### 服务端

- FastAPI 作为业务后端

### 基础设施

- Supabase Auth 负责认证
- Supabase Postgres 负责主数据库
- Cloudflare R2 负责对象存储

### 登录方式

- 手机号验证码登录
- Apple 登录

## 3. 为什么这样选

### 选择 FastAPI 的原因

- 这个项目的核心能力集中在 AI、OCR、文本抽取和结构化处理
- Python 生态更适合快速接入 AI 服务和文档处理链路
- MVP 阶段更需要迭代速度和开发效率
- FastAPI 结构清晰，适合后续继续拆分服务

### 暂不优先 Go 的原因

- 当前不是高并发事务型系统
- 首版重点不是吞吐极限，而是功能闭环和 AI 能力落地
- 在 Supabase 已经承担认证和数据库能力后，Go 的收益没有 FastAPI 明显

## 4. 总体架构

```text
Flutter App
  |-- Supabase Auth：登录、会话、用户身份
  |-- FastAPI：AI 解析、业务接口、文件上传签名、OCR 编排
  |-- Supabase Postgres：业务数据
  |-- Cloudflare R2：图片、PDF、报告文件
```

## 6. 推荐数据流

## 6.1 文本创建提醒

1. Flutter 提交自然语言输入
2. FastAPI 调用 AI 解析为结构化事件
3. FastAPI 返回结构化提醒草稿和置信信息
4. Flutter 在首页或日历页原地展示待确认提醒卡片
5. 用户确认、修改，或回答信息缺失时的追问
6. 用户确认后写入 Supabase Postgres
7. 客户端注册本地通知

产品交互上不把“AI 解析”暴露为用户必须点击的步骤。解析仍由后端完成，但前端应呈现为聊天式输入后的自动生成结果，避免“进入 AI 页 -> 点击解析 -> 跳表单 -> 保存”的多页面流程成为 MVP 主路径。

## 6.2 上传文档生成提醒

1. Flutter 请求 FastAPI 获取上传签名
2. Flutter 上传文件到 R2
3. FastAPI 保存文件元数据
4. FastAPI 执行 OCR 和 AI 结构化抽取
5. 返回候选提醒给客户端确认
6. 确认后写入事件表和文档表

## 6.3 记录健康数据

1. Flutter 提交指标数据
2. FastAPI 做规则校验
3. 写入 Supabase Postgres
4. 客户端刷新档案和趋势图

## 12. 风险与控制

### 风险 1：FastAPI 与 Supabase 权限边界不清晰

控制建议：

- 认证交给 Supabase
- 业务规则交给 FastAPI
- 数据权限通过用户 ID 和成员归属双重校验

### 风险 2：R2 文件上传链路复杂

控制建议：

- 统一通过 FastAPI 签名
- 所有文件元数据入库
- 删除文件时同步删除元数据

### 风险 3：AI 输出不稳定

控制建议：

- 统一 JSON schema
- 必须展示确认页
- 保留原始输入和置信度

## 14. MVP 安全架构（先上线可用版）

### 14.1 认证与鉴权

- 认证统一由 Supabase Auth 负责
- FastAPI 只信任已校验的 Supabase JWT
- 每个业务接口必须校验 user_id 与 member_id 归属关系
- 默认拒绝跨成员访问，避免家庭数据串读

### 14.2 传输与存储安全

- 全链路仅允许 HTTPS
- R2 文件访问采用短时签名 URL，不暴露长期公开链接
- 数据库中不保存明文敏感凭证（如第三方 API Key）
- 环境变量通过 .env 管理，不写入客户端代码仓库

### 14.3 MVP 安全基线

- 接口参数做基础校验（长度、类型、必填）
- 上传文件限制类型与大小（图片/PDF，建议 <= 10MB）
- 关键操作写审计日志（登录、上传、删除、成员编辑）

## 15. MVP API 规范（最小可执行）

### 15.1 基础约定

- Base Path：`/api/v1`
- 认证头：`Authorization: Bearer <supabase_access_token>`
- 请求体格式：JSON（文件上传接口除外）
- 时间统一使用 ISO 8601（含时区）

### 15.2 统一响应结构

成功响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

失败响应：

```json
{
  "code": 1001,
  "message": "member not found",
  "data": null
}
```

### 15.3 MVP 错误码建议

- 0：成功
- 1001：资源不存在
- 1002：参数错误
- 1003：未认证或 token 无效
- 1004：无权限访问该成员数据
- 2001：AI 解析失败
- 2002：OCR 处理失败
- 3001：文件上传失败

## 16. 第三方服务选型（MVP）

### 16.1 AI 服务

**AI 框架选型**
- 框架：LangChain
- 使用范围（V1）：
  - `ChatPromptTemplate` 管理提示词
  - `StructuredOutputParser` 解析结构化输出
  - `LLMChain` 简单编排
- 避免使用：复杂的 Agent、Memory 等高级功能（Phase 2/3 再引入）

**模型平台与选型**
- 主模型：Dashscope Qwen3.6-Plus
  - 多模态能力（文本+图像）
  - 支持结构化 JSON 输出
  - 中文时间表达解析能力强
  - 国内服务，延迟低（< 2秒）
  - 可能支持 OCR（需测试质量）
  
- 备选模型：
  - 阿里云百炼平台其他模型
  - OpenAI GPT-4（国际用户或降级）

**模型切换机制**
- 通过 LangChain 统一接口封装
- 环境变量配置模型选择：`AI_MODEL_PROVIDER`、`AI_MODEL_NAME`
- 降级策略：
  1. 主模型失败 → 自动切换备选模型
  2. 所有模型失败 → 回退到手动创建流程
  3. 记录失败日志供后续优化

**能力要求**
- 结构化 JSON 输出
- 中文时间表达解析（相对时间 + 绝对时间）
- 低延迟（< 3秒）
- 成本可控

### 16.2 OCR 服务

**主方案：百度 OCR**
- 通用文字识别（高精度版）
- 医疗票据识别（针对体检报告、检查单等）
- 当前代码已按百度 OCR 接入
- 价格需按当期官方套餐确认

**备选方案：iOS 原生 OCR**
- 使用场景：
  - 百度 OCR 服务不可用时降级
  - 离线场景
  - 用户隐私敏感场景
- 优势：免费、本地处理、隐私友好
- 劣势：准确率可能不如云服务、不支持 Android

**AI 二次解析流程**
1. 用户上传图片/PDF
2. 百度 OCR 提取文本（或 iOS 原生 OCR）
3. 保存 OCR 原文到数据库
4. Qwen3.6-Plus 从文本中提取结构化信息：
   - 复查建议（如"3个月后复查"）
   - 时间信息
   - 医院/科室
   - 检查项目
   - 异常指标
5. 返回候选提醒列表供用户确认
6. 文件归档保存到 R2

**Qwen 多模态 OCR 测试**
- Qwen3.6-Plus 支持图像输入，可能直接完成 OCR + 解析
- 建议先测试质量：
  - 如果质量足够 → 简化为单一 Qwen 调用
  - 如果质量不足 → 保持百度 OCR + Qwen 解析的两步流程

**原文保留策略**
- 必须保存 OCR 原文，便于用户核对
- 支持用户手动编辑识别结果
- 提供"重新识别"功能

### 16.3 短信与推送

- 手机验证码：使用 Supabase Auth 提供的短信链路
- 提醒通知：MVP 仅 iOS 本地通知，不依赖服务端推送

### 16.4 RAG 功能规划（非 V1）

**V1 范围说明**
- V1 不包含 RAG（检索增强生成）功能
- V1 专注核心闭环：提醒创建、文档上传、OCR 解析、健康档案

**RAG 适用场景（Phase 2/3）**
- 健康文档智能检索（"上次甲状腺检查结果是什么？"）
- 历史报告问答（"过去一年血糖趋势如何？"）
- 个性化健康时间线总结

**架构预留设计**
- 文档入库时预留向量化字段（如 `embedding_vector`）
- FastAPI 预留查询接口：`/api/v1/ai/query`
- LangChain 可无缝扩展到 RAG：
  - Retriever
  - Vector Store
  - QA Chain

**后续技术建议**
- 向量数据库：Supabase pgvector（优先）
- 分块策略：按段落/检查项目切分文档
- 检索策略：语义检索 + 关键词检索混合

## 17. 环境配置（MVP）

### 17.1 环境划分

- dev：本地开发环境
- staging：联调与预发布验证
- prod：线上环境

### 17.2 关键环境变量

- `SUPABASE_URL`
- `SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`（仅服务端）
- `R2_ACCESS_KEY_ID`
- `R2_SECRET_ACCESS_KEY`
- `R2_BUCKET`
- `AI_API_KEY`

### 17.3 配置原则

- 客户端仅持有 anon key，不持有 service role key
- 所有密钥按环境隔离
- 生产环境密钥定期轮换

## 18. 部署方案（MVP）

### 18.1 FastAPI 部署

- 使用 Docker 部署 FastAPI
- 运行方式：`uvicorn + gunicorn`（多 worker）
- 通过反向代理（Nginx/Caddy）提供 HTTPS 入口

### 18.2 发布流程（最小版）

1. 合并到主分支
2. 自动化执行基础测试
3. 构建镜像并发布到 staging
4. 验证核心流程后发布到 prod

### 18.3 回滚策略

- 保留最近可用镜像版本
- 发布失败时直接回滚到上一个稳定版本
- 数据库变更采用可回滚迁移脚本

## 19. 监控与告警（MVP）

### 19.1 必监控指标

- API 成功率
- API P95 响应时间
- AI/OCR 调用失败率
- 文件上传失败率

### 19.2 日志要求

- 接口请求日志（trace_id、user_id、path、status）
- 错误日志（异常堆栈、上下文参数）
- 审计日志（成员删除、文件删除、登录）

### 19.3 MVP 告警规则

- 5 分钟内接口错误率 > 5% 触发告警
- AI/OCR 连续失败超过阈值触发告警
- 文件上传失败率异常升高触发告警
