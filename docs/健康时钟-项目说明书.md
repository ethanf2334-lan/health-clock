# 健康时钟 App 项目说明书

## 1. 文档目标

本文档用于指导健康时钟 App 的产品设计、技术方案、研发拆解和 AI 辅助开发协作，适合作为后续生成原型、页面、数据库、接口和代码时的基础输入文档。

## 2. 项目概述

### 2.1 项目名称

健康时钟

### 2.2 项目定位

一款基于 AI 的家庭健康提醒与健康档案管理 App，支持通过自然语言、语音和医疗文档识别来自动生成健康日程，并沉淀个人及家庭成员的健康资料。

### 2.3 首期平台

- 客户端：Flutter
- 首发平台：iOS
- 后续扩展：Android

## 3. 项目目标

### 产品目标

- 以 AI 为核心输入方式，降低健康提醒创建门槛
- 形成“提醒 + 档案 + 文档 + 数据记录”的统一健康管理体验
- 满足单用户和家庭多成员管理场景

### 技术目标

- 用 Flutter 搭建可持续演进的跨平台架构
- 采用模块化设计，便于后期扩展 Android 和更多 AI 能力
- 保留本地能力与云端能力的平衡，提升稳定性和上线效率

- MedicalDocument：医疗/体检文档

- NotificationTask：通知任务

### 实体关系

- 一个 User 对应多个 ProfileMember
- 一个 ProfileMember 对应多个 HealthEvent
- 一个 ProfileMember 对应多个 MedicalDocument
- 一个 ProfileMember 对应多个 HealthMetricRecord
- 一个 HealthEvent 可关联一个或多个 MedicalDocument

## 8. 功能设计原则

### 原则 1：AI 为主，手动兜底

AI 是核心体验，但不能成为唯一入口。任何 AI 生成结果都必须允许用户确认和修改。

### 原则 2：面向家庭场景

成员切换必须足够显眼、简单，避免误把父母资料记到自己名下。

### 原则 3：数据沉淀优先

提醒不是孤立事件，应尽量和成员、报告、健康指标形成关联。

### 原则 4：合规表达

产品只做健康管理，不做医学诊断。

## 9.2 服务端建议

V1 建议采用“Supabase + FastAPI”的组合方案，兼顾开发效率、AI 能力接入和后续扩展空间。

### 推荐架构

- 认证：Supabase Auth
- 登录方式：手机号验证码登录 + Apple 登录
- 数据库：Supabase Postgres
- 对象存储：Cloudflare R2
- 业务 API：FastAPI
- AI/OCR 编排：FastAPI 服务负责调用与整合
- 客户端：Flutter 直接连接 Supabase 与业务后端

### 为什么优先 FastAPI

- 更适合接入 AI、OCR、文档解析与文本处理能力
- Python 生态对结构化抽取、异步任务和原型验证更友好
- 对当前 MVP 阶段来说，开发效率优先于极致性能
- 后续如果业务变重，仍可以把部分高并发模块逐步拆到 Go

### 为什么当前不优先 Go

- 你当前的核心复杂度在 AI 解析和文档处理，不在超高并发 API
- 如果首版直接用 Go，开发成本与迭代门槛会更高
- 在 Supabase 已承担认证和数据库基础设施的前提下，Go 的优势暂时发挥不充分

### Supabase 负责内容

- 认证
- 用户与会话管理
- 手机号验证码登录
- Apple 登录
- Postgres 数据库
- Row Level Security 权限隔离

### FastAPI 负责内容

- AI 提醒解析
- OCR 结果二次结构化
- 健康文档摘要
- 候选提醒生成
- 业务规则处理
- R2 上传签名和文件访问控制
- 与 Supabase 数据联动

### Cloudflare R2 负责内容

- 存储体检报告
- 存储就医图片
- 存储 PDF 和其他健康资料文件
- 通过 S3 兼容接口对接后端

### V1 推荐

如果目标是尽快上线并验证产品，推荐方案为：

- Flutter
- Supabase Auth
- Supabase Postgres
- Cloudflare R2
- FastAPI

## 9.3 AI 能力建议

### AI 框架

- 框架：LangChain
- V1 使用范围：
  - `ChatPromptTemplate` 管理提示词
  - `StructuredOutputParser` 解析结构化输出
  - `LLMChain` 简单编排
- 后续扩展：Agent、Memory、RAG

### 模型平台

- 主模型：Dashscope Qwen3.6-Plus
  - 多模态能力（文本+图像）
  - 支持结构化 JSON 输出
  - 中文时间表达解析能力强
  - 国内服务，延迟低
  
- 备选模型：
  - 阿里云百炼平台其他模型
  - OpenAI GPT-4（国际用户）

### 模型切换设计

- 通过 LangChain 统一接口封装
- 环境变量配置模型选择
- 降级策略：主模型失败 → 备选模型 → 手动创建

### AI 使用点

- 文本意图解析
- 语音转写后的提醒生成
- OCR 文本摘要
- 文档中复查建议抽取
- 健康事件结构化

### AI 输出建议统一为结构化 JSON

示例字段：

```json
{
  “member_name”: “妈妈”,
  “event_title”: “甲状腺复查”,
  “event_type”: “follow_up”,
  “scheduled_at”: “2026-06-30T09:00:00+08:00”,
  “is_all_day”: false,
  “repeat_rule”: null,
  “source”: “ai_text”,
  “confidence”: 0.92,
  “needs_confirmation”: true
}
```

### AI 设计原则

- 不直接落库原始自然语言结果
- 统一经结构化解析后保存
- 置信度低时必须要求人工确认
- 提示词需明确禁止”医学建议扩写”

## 9.4 OCR 方案建议

### V1 主方案

- 腾讯云 OCR（通用文字识别 + 医疗票据识别）
- AI 二次解析：Qwen3.6-Plus 提取结构化信息

### 备选方案

- iOS 原生 OCR（降级或离线场景）

### 建议策略

- 图片/PDF 先做 OCR
- OCR 文本再交给 AI 做结构化抽取
- 原文件、OCR 文本、结构化摘要分层存储
- 必须保留 OCR 原文供用户核对

### Qwen 多模态 OCR 评估

- Qwen3.6-Plus 具备图像输入能力，可尝试直接 OCR + 解析
- 先做 A/B 测试：
  - 路径 A：腾讯云 OCR + Qwen 解析
  - 路径 B：Qwen 直接多模态解析
- 以准确率和成本决定最终主链路

## 15. 风险点与应对建议

### 风险 1：AI 时间解析不稳定

应对：

- 增加确认页
- 保留手动编辑入口
- 记录原始输入文本

### 风险 2：OCR 识别质量不稳定

应对：

- 支持用户查看 OCR 原文
- 支持手动修正
- 区分“原文件”和“AI 提取结果”

### 风险 3：健康数据涉及隐私

应对：

- 强化权限与存储安全
- 在产品中提供隐私说明
- 设计删除与导出能力

### 风险 4：家庭成员数据混淆

应对：

- 成员切换要强提示
- 新建提醒时默认显示当前成员
- 保存前二次展示归属成员

## 19. MVP 成本估算

### 19.1 基础设施成本（月度）

**Supabase（按用户规模）**
- 免费额度：500MB 数据库、1GB 文件存储、50,000 月活用户
- Pro 版（$25/月）：8GB 数据库、100GB 文件存储、100,000 月活用户
- MVP 阶段建议：先用免费版，用户超 500 人后升级 Pro

**Cloudflare R2**
- 存储：$0.015/GB/月
- 下载流量：免费（无出站费用）
- 预估：100 用户 × 平均 50MB 文档 = 5GB ≈ $0.08/月
- MVP 阶段成本：< $1/月

**AI 服务（按调用量）**
- 主模型：Dashscope Qwen3.6-Plus
- 文本解析：预估每用户每月 10 次 × 100 用户 = 1000 次
- 文档解析：预估每用户每月 3 次 × 100 用户 = 300 次
- Dashscope 定价参考（需确认最新价格）：
  - Qwen3.6-Plus：约 ¥0.02/千tokens（输入）+ ¥0.06/千tokens（输出）
  - 预估单次调用成本：¥0.05-0.15
- 成本估算：约 ¥65-195/月（1300 次调用）

**OCR 服务**
- 腾讯云 OCR：约 ¥0.15/次
- 预估：100 用户 × 3 次/月 = 300 次
- 成本估算：约 ¥45/月

**短信验证码**
- 国内短信：约 ¥0.05/条
- 预估：100 用户 × 2 次登录/月 = 200 条 ≈ ¥10/月

**Apple Developer**
- $99/年（约 $8.25/月）

**MVP 阶段总成本预估：¥120-250/月 + $8.25/月（不含人力）**
