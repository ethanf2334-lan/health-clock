# 健康时钟 MVP 项目进度总览

更新时间：2026-04-24

本文档基于以下资料交叉整理：

- 产品文档：`docs/健康时钟-需求文档.md`
- 项目说明：`docs/健康时钟-项目说明书.md`
- 技术架构：`docs/健康时钟-技术架构文档.md`
- 实际代码：`health-clock-backend/` 与 `health-clock-app/`

目标：对照 MVP 范围，明确哪些已经实现，哪些部分实现，哪些尚未实现，方便后续排期与收尾。

## 1. 总体判断

当前项目状态可以概括为：

- 后端 MVP 主链路大体已具备，核心 API、数据库迁移、AI 文本解析、OCR、文件上传、成员管理、提醒管理、健康指标记录都已经有实际代码。
- 前端仍处于“骨架 + 少量可用页面”阶段，只有 AI 文本解析页面、成员列表/新增/删除接入了后端，登录、日历、文档、健康指标、提醒创建确认、成员切换等关键页面尚未完成。
- 如果以“后端能力是否存在”来判断，MVP 已完成约 70%。
- 如果以“用户是否能在 App 中完成完整 MVP 闭环”来判断，MVP 更接近 40% 到 50%，主要瓶颈在 Flutter 前端。

## 3. 已实现内容

### 3.1 后端已实现

以下功能在后端代码中已有实际实现，不是纯占位：

- 基础 FastAPI 服务启动与路由注册
- Supabase token 鉴权和 `/auth/me` 用户信息获取
- 成员管理 CRUD
- 健康提醒 CRUD、筛选查询、标记完成
- AI 文本解析接口 `/ai/parse-text`
- 文档上传签名接口、文档元数据 CRUD
- 百度 OCR 识别与 OCR 后 AI 二次提取
- 候选提醒抽取
- 健康指标记录的新增、查询、删除
- 初始数据库迁移脚本、RLS 策略、关联表设计

关键代码位置：

- 后端入口：`health-clock-backend/app/main.py`
- 成员接口：`health-clock-backend/app/api/v1/members.py`
- 提醒接口：`health-clock-backend/app/api/v1/events.py`
- AI 解析：`health-clock-backend/app/api/v1/ai.py`
- 文档/OCR：`health-clock-backend/app/api/v1/documents.py`
- 指标记录：`health-clock-backend/app/api/v1/metrics.py`
- 数据库迁移：`health-clock-backend/migrations/001_initial_schema.sql`

## 4. 部分实现但未闭环的内容

这些模块已经有一半以上基础，但还不能算真正完成。

### 4.1 登录与用户态

现状：

- 后端可以校验 Supabase token。
- 前端没有手机号验证码登录流程。
- 前端没有 Apple 登录流程。
- 前端没有真正的登录态管理，当前通过“跳过登录（测试模式）”进入首页。

影响：

- 用户体系在 MVP 中仍未打通。
- 成员、提醒、文档等真实用户数据链路无法通过 App 完整验证。

### 4.2 成员管理

现状：

- 后端成员 CRUD 已完成。
- 前端成员列表、新增、删除可用。
- 成员编辑入口是 TODO。
- 成员详情页没有实现。
- 成员切换机制没有实现。

影响：

- 目前只能做最基础的成员维护，还不能支持文档中要求的家庭场景切换体验。

### 4.3 AI 创建提醒

现状：

- 前端 AI 输入页已能请求后端解析文本。
- 页面能展示标题、类型、时间、置信度和确认提示。
- “创建提醒”按钮仍是占位提示，未把解析结果写入提醒表。

影响：

- 文档强调的核心卖点“AI 一句话创建提醒”还没有在 App 中闭环。

### 4.4 健康提醒管理

现状：

- 后端提醒创建、更新、删除、查询、完成接口都有。
- 前端只有 `EventRepository`，没有提醒列表页、详情页、创建页、编辑页。
- `CalendarScreen` 还是占位页面。

影响：

- 虽然后端能力具备，但用户还不能在 App 中查看或管理提醒。

### 4.5 文档上传与 OCR

现状：

- 后端支持 R2 预签名上传、文档元数据保存、OCR 处理、候选提醒提取。
- 前端没有文件选择、上传、文档列表、OCR 结果展示、候选提醒确认等页面。

影响：

- 文档上传是 MVP 的关键链路之一，但当前几乎只完成了服务端。

### 4.6 健康指标

现状：

- 后端支持记录新增、查询、删除。
- 前端没有录入表单、历史列表、趋势图表。

影响：

- 文档里提到的“健康指标记录与趋势展示”还没有进入可体验状态。

### 4.7 通知

现状：

- 前端通知服务已封装，权限页也存在。
- 但没有把提醒创建与本地通知调度绑定起来。
- 通知点击回跳还是 TODO。
- “去设置”按钮尚未真正打开系统设置。

影响：

- 通知能力有基础代码，但还没有进入完整业务链路。

## 6. 文档与代码不一致的地方

### 6.1 OCR 供应商

- 根 README 曾出现“百度 OCR”和“腾讯云 OCR”两种说法。
- 实际代码是百度 OCR：
  - `health-clock-backend/app/services/ocr_service.py`
  - `health-clock-backend/app/core/config.py`

结论：

- 项目实际实现应以“百度 OCR”为准。
- 文档里仍出现“腾讯云 OCR”的地方应统一修正。

### 6.2 前端 README 进度描述偏旧

- `health-clock-app/README.md` 的“下一步”仍写着“配置 Supabase 认证、实现登录、实现成员管理”等较早期表述。
- 实际代码中成员列表、新增、删除与 AI 解析页面已经存在。

结论：

- 前端 README 可以更新为更贴近当前状态的版本。

## 8. 建议的收尾优先级

如果目标是尽快完成 MVP，建议按下面顺序推进：

1. 打通真实登录
2. 完成提醒闭环
3. 完成文档上传闭环
4. 完成成员切换
5. 完成健康指标最小闭环
6. 接通本地通知
7. 清理测试和基础工程问题

具体建议如下：

### P0：必须优先完成

- Flutter 登录流程接 Supabase
- AI 解析结果确认后创建提醒
- 提醒列表页或最小日历列表页
- 手动创建提醒页
- 文档上传页
- OCR 结果与候选提醒确认页
- 成员切换

### P1：建议在 MVP 内完成

- 成员编辑
- 健康指标录入与历史页
- 通知权限引导接入业务流程
- 本地通知调度
- “我的”页面基础信息

### P2：可以在 MVP 后半段补齐

- 趋势图
- 更完整的设置页
- 通知点击跳转
- 更细的筛选与体验优化

## 9. 当前 MVP 完成度结论

按“文档要求 + 实际代码 + 是否形成用户闭环”综合判断：

- 后端完成度：高
- 前端完成度：中低
- 整体 MVP 完成度：中等偏下

一句话总结：

后端主能力已经基本铺好，接下来最关键的工作不是继续扩后端，而是集中完成 Flutter 端的登录、提醒、文档、成员切换这四条主链路，把现有能力真正串成可交付的 MVP。
